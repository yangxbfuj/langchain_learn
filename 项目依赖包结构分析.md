## 2. langchain_core — 核心基础包

> **这是学习的起点。** 一切 LangChain 组件都实现了 `langchain_core` 中定义的接口。

### 2.1 Runnable 协议（`runnables/`）— LCEL 核心

**这是整个框架的心脏。** 每个 LangChain 组件都是一个 `Runnable`。

```
Runnable[Input, Output]
    │
    ├── invoke(input) -> Output           # 同步执行
    ├── ainvoke(input) -> Output          # 异步执行
    ├── batch(inputs) -> list[Output]     # 批量执行
    ├── stream(input) -> Iterator[Output] # 流式执行
    ├── astream(input) -> AsyncIterator   # 异步流式
    └── 支持 | 管道操作符组合             # prompt | model | parser
```

**核心类：**

| 类 | 作用 |
|----|------|
| `Runnable` | 抽象基类，所有组件的统一接口 |
| `RunnableSequence` | 通过 `\|` 创建的链式序列 |
| `RunnableParallel` / `RunnableMap` | 并行执行多个 Runnable |
| `RunnableLambda` | 将普通函数包装为 Runnable |
| `RunnableBranch` | 条件分支（if/elif/else） |
| `RunnablePassthrough` | 透传输入，可配合 `assign` 添加字段 |
| `RunnableWithFallbacks` | 失败后回退到备用方案 |
| `RunnableWithMessageHistory` | 自动管理对话历史 |
| `RunnableBinding` | 绑定参数（如 `bind_tools()`） |
| `RunnableConfig` | 贯穿调用链的配置（callbacks, tags, metadata...） |

**LCEL 示例：**
```python
chain = prompt | model | StrOutputParser()
result = chain.invoke({"topic": "AI"})
```

## 3. langchain — 智能体框架

> v1.x 的新一代智能体 API。核心就是 `create_agent()` + Middleware。

### 3.1 包结构

```
langchain/
├── agents/                      # 智能体核心
│   ├── factory.py               # ★ create_agent() — 创建智能体的主入口
│   ├── structured_output.py     # 结构化输出策略（ToolStrategy/ProviderStrategy/AutoStrategy）
│   ├── _subagent_transformer.py # 子智能体流追踪
│   └── middleware/               # ★ 中间件系统（14个内置中间件）
│       ├── types.py             #    AgentMiddleware 基类 + 生命周期钩子
│       ├── model_fallback.py    #    模型回退
│       ├── model_retry.py       #    模型重试
│       ├── model_call_limit.py  #    模型调用次数限制
│       ├── tool_retry.py        #    工具重试
│       ├── tool_call_limit.py   #    工具调用次数限制
│       ├── tool_emulator.py     #    工具模拟器（测试用）
│       ├── tool_selection.py    #    工具选择（减少发送给 LLM 的工具列表）
│       ├── summarization.py     #    对话摘要（超长上下文处理）
│       ├── context_editing.py   #    上下文编辑（清除旧工具输出）
│       ├── human_in_the_loop.py #    人在回路（审查/批准工具调用）
│       ├── pii.py               #    PII 检测与脱敏
│       ├── shell_tool.py        #    Shell 工具（带沙箱策略）
│       ├── todo.py              #    任务清单工具
│       ├── file_search.py       #    文件搜索工具（Glob/Grep）
│       └── provider_tool_search.py # 提供商原生工具搜索
├── chat_models/base.py          # init_chat_model() — 统一模型初始化
├── embeddings/base.py           # init_embeddings() — 统一 Embedding 初始化
├── messages/                    # 消息类型（从 langchain_core 重导出）
├── tools/                       # 工具类型（从 langchain_core 重导出）
└── rate_limiters/               # 限流器（从 langchain_core 重导出）
```

### 3.2 `create_agent()` — 核心 API

```python
from langchain.agents import create_agent

agent = create_agent(
    model="openai:gpt-5.5",         # "provider:model" 格式
    system_prompt="You are helpful.",
    tools=[my_tool],                 # 工具列表
    middleware=[],                   # 中间件
    checkpointer=checkpointer,       # 状态持久化
    store=store,                     # 长期记忆
    response_format=MySchema,        # 结构化输出
)
```

**内部流程：**
1. `init_chat_model()` 将字符串转为 `BaseChatModel` 实例
2. 系统提示词标准化为 `SystemMessage`
3. 工具注册到 `ToolNode`
4. 中间件钩子发现和组合（洋葱模型）
5. 构建 `StateGraph` → `compile()` → 可执行的 `Pregel` 图
6. 智能体循环：model → tools → model → ... → end

### 3.3 Middleware 中间件系统

每个中间件通过覆写 `AgentMiddleware` 的钩子方法介入智能体生命周期：

```
                    before_agent()
                         │
                    ┌────▼────┐
              ┌────>│  model  │────┐
              │     └────┬────┘    │
              │     before_model() │ after_model()
              │          │         │
              │     wrap_model_call() ── 模型调用拦截
              │          │         │
              │     ┌────▼────┐    │
              │     │  tools  │    │
              │     └────┬────┘    │
              │     wrap_tool_call() ── 工具调用拦截
              │          │         │
              └──────────┘         │
                         │
                    after_agent()
```

**14 个内置中间件速查：**

| 中间件 | 功能 | 关键参数 |
|--------|------|----------|
| `ModelRetryMiddleware` | 模型调用失败重试 | `max_retries`, `backoff_factor` |
| `ModelFallbackMiddleware` | 模型失败后切换备用模型 | `fallbacks: list[str]` |
| `ModelCallLimitMiddleware` | 限制模型调用次数 | `run_limit`, `thread_limit` |
| `ToolRetryMiddleware` | 工具调用失败重试 | `max_retries`, `backoff_factor` |
| `ToolCallLimitMiddleware` | 限制工具调用次数 | `run_limit`, `thread_limit` |
| `SummarizationMiddleware` | 超长上下文自动摘要 | `trigger`（token 阈值）, `keep` |
| `ContextEditingMiddleware` | 清除旧工具输出来释放上下文 | `trigger`, `clear_at_least` |
| `HumanInTheLoopMiddleware` | 人在回路审批工具调用 | 使用 `interrupt()` 机制 |
| `PIIMiddleware` | 检测和脱敏 PII | 策略：block/redact/mask/hash |
| `ShellToolMiddleware` | 提供持久 Shell 工具 | 执行策略：Host/CodexSandbox/Docker |
| `TodoListMiddleware` | 任务规划工具 `write_todos` | 自动注入系统提示 |
| `ToolEmulator` | 用 LLM 模拟工具执行（测试） | 可指定模拟的工具 |
| `ToolSelectionMiddleware` | 用 LLM 筛选最相关的工具 | 减少发送给主模型的工具数 |
| `ProviderToolSearchMiddleware` | 将工具搜索委托给模型提供商 | Anthropic/OpenAI 原生支持 |

**自定义中间件也很简单：**
```python
from langchain.agents.middleware import AgentMiddleware

class MyMiddleware(AgentMiddleware):
    def before_model(self, state, runtime):
        # 每次调用模型前执行
        ...
        return {"messages": [SystemMessage(content="custom context")]}
```

**装饰器方式创建中间件：**
```python
from langchain.agents.middleware import before_model, after_model

@before_model
def my_hook(state, runtime):
    # 在每次模型调用前执行
    return {"messages": [...]}
```

### 3.4 `init_chat_model()` — 统一模型初始化

支持 27 个内置提供商映射：

```python
from langchain.chat_models import init_chat_model

# 显式指定提供商
model = init_chat_model("openai:gpt-5.5")
model = init_chat_model("anthropic:claude-sonnet-5")
model = init_chat_model("ollama:llama3")

# 自动推断提供商（通过模型名前缀）
model = init_chat_model("gpt-4.5")     # → openai
model = init_chat_model("claude-4.8")  # → anthropic
```

**支持的提供商：** OpenAI, Anthropic, Azure, AWS Bedrock, Google VertexAI/GenAI, Cohere, Fireworks, Mistral, HuggingFace, Groq, Ollama, DeepSeek, IBM Watsonx, NVIDIA, xAI, OpenRouter, Perplexity 等 27 个

### 3.5 子智能体支持

当智能体以工具形式调用另一个命名智能体时，`SubagentTransformer` 会：
- 检测子智能体边界（通过 `lc_agent_name` 追踪）
- 创建独立的流 Mux 通道
- 提供 `SubagentRunStream` 句柄供独立消费

## 4. langgraph — 图执行框架

> 构建有状态的、多角色的 LLM 应用。灵感来自 Google Pregel 的 BSP（Bulk Synchronous Parallel）模型。

### 4.1 包结构总览

```
langgraph/
├── graph/            # ★ StateGraph 构建 API（用户入口）
├── pregel/           # ★ Pregel 运行时引擎（BSP 执行循环）
├── channels/         # 状态存储原语（10 种 Channel 类型）
├── checkpoint/       # 持久化层（状态快照）
├── prebuilt/         # 预构建组件（ToolNode, create_react_agent）
├── func/             # 函数式 API（@entrypoint, @task）
├── managed/          # 托管值（动态注入 IsLastStep, RemainingSteps）
├── stream/           # v3 流式协议
├── cache/            # 缓存层
├── store/            # 长期记忆（跨线程持久化 KV 存储）
├── _internal/        # 内部实现（不公开）
├── utils/            # 工具函数
├── types.py          # 核心公共类型（Send, Command, Interrupt, RetryPolicy...）
├── errors.py         # 错误类型（GraphRecursionError, GraphInterrupt...）
├── constants.py      # 常量（START, END, TAG_NOSTREAM, TAG_HIDDEN）
├── config.py         # 运行时访问器（get_config(), get_store(), get_stream_writer()）
├── runtime.py        # 运行时上下文
├── callbacks.py      # 图生命周期回调
└── warnings.py       # 废弃警告
```

### 4.2 `graph/` — 图构建 API（用户入口）

**核心工作流：**

```python
from langgraph.graph import StateGraph, MessagesState, START, END

# 1. 定义状态 Schema
class MyState(TypedDict):
    messages: Annotated[list[AnyMessage], add_messages]

# 2. 构建图
builder = StateGraph(MyState)

# 3. 添加节点
builder.add_node("agent", call_model)
builder.add_node("tools", ToolNode(tools))

# 4. 添加边
builder.add_edge(START, "agent")
builder.add_conditional_edges("agent", tools_condition)
builder.add_edge("tools", "agent")

# 5. 编译执行
graph = builder.compile(checkpointer=memory)
result = graph.invoke({"messages": [HumanMessage("hello")]})
```

**核心类：**

| 类/类型 | 作用 |
|---------|------|
| `StateGraph[StateT, ContextT, InputT, OutputT]` | 图构建器，定义节点、边、状态 Schema |
| `CompiledStateGraph`（即 `Pregel`） | `.compile()` 的产物，可执行的图 |
| `add_messages` | 消息列表合并 reducer（去重、RemoveMessage 墓碑机制） |
| `MessagesState` | 预定义消息状态：`{messages: Annotated[list, add_messages]}` |
| `START` / `END` | 图的入口和出口哨兵值 |
| `Send` | 动态并行分支：向多个节点发送不同状态 |
| `Command` | 节点可返回的指令（状态更新 + 路由指令） |
| `Interrupt` | 中断执行等待人工输入 |

**State Schema 中的 Channel 类型推导：**

| 注解 | Channel 类型 | 行为 |
|------|-------------|------|
| `Annotated[T, operator.add]` | `BinaryOperatorAggregate` | 累加 |
| `Annotated[T, add_messages]` | `BinaryOperatorAggregate` | 消息合并 |
| `Annotated[T, ManagedValue]` | 托管值 | 动态注入 |
| `Annotated[T, None]`（默认） | `LastValue` | 仅保留最后值 |

### 4.3 `pregel/` — 运行时引擎（BSP 模型）

**这是 LangGraph 的执行核心。** 灵感来自 Google Pregel 的大规模图处理系统。

**Pregel 超步（Superstep）循环：**

```
┌──────────────────────────────────────────┐
│ 1. 加载 Checkpoint（如果有）               │
│ 2. prepare_next_tasks() — 确定要运行的节点 │
│ 3. PregelRunner 执行任务（并发+重试+超时） │
│ 4. apply_writes() — 更新 Channel 值       │
│ 5. 创建 Checkpoint                       │
│ 6. 重复直到无任务或达到递归限制            │
└──────────────────────────────────────────┘
```

**核心类：**

| 类 | 作用 |
|----|------|
| `Pregel` | 编译后的可执行图（`StateGraph.compile()` 的产物） |
| `PregelLoop`（sync/async） | BSP 超步循环实现 |
| `PregelRunner` | 任务执行器（并发、重试、超时、缓存） |
| `ChannelRead` | 从共享状态读取 Channel 值 |
| `ChannelWrite` | 向 Channel 写入值（支持 Send 动态路由） |
| `NodeBuilder` | 构建 Pregel 节点（绑定触发器和策略） |

**Pregel 的关键方法：**

| 方法 | 作用 |
|------|------|
| `invoke(input, config)` / `ainvoke()` | 执行一次，返回最终输出 |
| `stream(input, config, stream_mode=...)` | 流式执行 |
| `stream_events(input, config, version="v3")` | v3 流式协议 |
| `update_state(config, values)` | 手动更新状态 |
| `get_state(config)` | 查看当前状态 |
| `get_state_history(config)` | 列出历史 Checkpoint（时间旅行） |
| `get_graph(config)` | 获取图结构用于可视化 |

### 4.4 `channels/` — 状态存储原语

每个状态 Key 对应一个 Channel 实例。Channel 决定了该 Key 的读写行为。

| Channel | 行为 |
|---------|------|
| **`LastValue`** | 每步仅保留一个最后写入的值，多写报错 |
| **`LastValueAfterFinish`** | 类似于 `LastValue`，但值仅在 `finish()` 后可见 |
| **`AnyValue`** | 保留最后写入的值，允许多写（假设值相同） |
| **`BinaryOperatorAggregate`** | 使用 reducer 函数累加（如 `operator.add`） |
| **`Topic`** | Pub/Sub 模式，支持跨步骤累积或每步清空 |
| **`EphemeralValue`** | 仅保存上一步的值，消耗后清空 |
| **`NamedBarrierValue`** | 等待所有命名值到达后才可用 |
| **`UntrackedValue`** | 像 `LastValue` 但不持久化 |
| **`DeltaChannel`**（beta） | 仅存储哨兵值，通过回放祖先写入来重建状态 |

### 4.5 `checkpoint/` — 持久化与时间旅行

| 组件 | 作用 |
|------|------|
| `BaseCheckpointSaver` | 持久化抽象基类 |
| `Checkpoint` | 状态快照：channel_values, channel_versions, versions_seen |
| `InMemorySaver` / `MemorySaver` | 内存实现（可选 `PersistentDict` 持久化） |
| `JsonPlusSerializer` | 默认序列化（JSON + pickle 回退） |
| `EncryptedSerializer` | 加密序列化 |

**支持的操作：**
- `get(config)` — 读取 Checkpoint
- `put(config, checkpoint)` — 保存 Checkpoint
- `list(config)` — 列出历史 Checkpoint → **时间旅行**
- `delete_thread(thread_id)` — 删除线程
- `copy_thread(source, target)` — 复制线程

### 4.6 `prebuilt/` — 预构建组件

| 组件 | 作用 |
|------|------|
| `create_react_agent(model, tools)` | 一行代码创建 ReAct 模式的智能体图 |
| `ToolNode(tools)` | 预构建的工具执行节点（支持并行、错误处理） |
| `tools_condition` | 条件边函数：有工具调用→tools，否则→END |
| `ValidationNode(tools)` | 验证工具调用（不实际执行） |
| `InjectedState` | 依赖注入标记：工具参数自动获取图状态 |
| `InjectedStore` | 依赖注入标记：工具参数自动获取持久化存储 |
| `ToolRuntime` | 工具运行时注入（state + context + config + stream_writer） |

### 4.7 `func/` — 函数式 API

不需要显式构建 StateGraph 的替代 API：

```python
from langgraph.func import entrypoint, task

@task
def my_task(x: str) -> str:
    return f"processed: {x}"

@entrypoint(checkpointer=memory)
def my_workflow(input_data: dict) -> dict:
    # 可以并行调用多个 task
    futures = [my_task(item) for item in input_data["items"]]
    results = [f.result() for f in futures]
    return {"results": results}

# 和 StateGraph 一样使用
result = my_workflow.invoke({"items": ["a", "b", "c"]})
```

### 4.8 `stream/` — v3 流式协议

**事件流：** 图发出 v2 StreamPart → `convert_to_protocol_event` → `StreamMux` → 变换器管道 → `StreamChannel` → 消费者

| 组件 | 作用 |
|------|------|
| `GraphRunStream` | `stream_events(version="v3")` 的返回值 |
| `StreamMux` | 中心事件分发器 |
| `StreamTransformer` | 自定义流变换的扩展点 |
| `StreamChannel` | 单消费者队列 |

**内置变换器：** `ValuesTransformer`, `MessagesTransformer`, `UpdatesTransformer`, `DebugTransformer`, `CheckpointsTransformer`, `TasksTransformer`, `SubgraphTransformer`, `LifecycleTransformer`, `CustomTransformer`

### 4.9 `store/` — 长期记忆

跨线程、跨对话的持久化 KV 存储，支持语义搜索。

| 组件 | 作用 |
|------|------|
| `BaseStore` | 抽象存储接口 |
| `InMemoryStore` | 内存实现（可选 NumPy 余弦相似度搜索） |
| 操作 | `put`, `get`, `delete`, `search`（语义搜索）, `list_namespaces` |
| `IndexConfig` | 向量搜索配置：dims, embed, fields |

### 4.10 `cache/` — 缓存层

| 实现 | 用途 |
|------|------|
| `BaseCache` | 抽象接口 |
| `InMemoryCache` | 线程安全内存缓存（TTL 支持） |
| `RedisCache` | Redis 缓存 |

## 7. 架构总结与学习路线

### 7.1 技术栈分层

```
┌─────────────────────────────────────────┐
│         应用层                            │
│  你的 Agent / Chain / Workflow            │
├─────────────────────────────────────────┤
│         Orchestration 层                  │
│  langchain (create_agent + middleware)    │
│  langgraph (StateGraph + Pregel)         │
├─────────────────────────────────────────┤
│         抽象层                            │
│  langchain_core (Runnable, Messages,     │
│  Tools, Prompts, Callbacks...)           │
├─────────────────────────────────────────┤
│         集成层                            │
│  langchain_openai, langchain_community,  │
│  langchain_mcp_adapters, ...             │
└─────────────────────────────────────────┘
```

### 7.2 推荐学习路线

```
第 1 步：langchain_core（基础）
  ├── 理解 Runnable 协议（Runnable, |, invoke, stream, batch）
  ├── 掌握消息类型（HumanMessage, AIMessage, SystemMessage, ToolMessage）
  ├── 学会使用 Prompt 模板（ChatPromptTemplate, MessagesPlaceholder）
  └── 了解 Tool 系统（@tool 装饰器, BaseTool）

第 2 步：langgraph（图执行）
  ├── 理解 StateGraph 构建（add_node, add_edge, add_conditional_edges）
  ├── 掌握 State Schema（TypedDict + Annotated + reducer）
  ├── 了解 Channel 类型和 checkpoint 持久化
  ├── 学会使用 ToolNode 和工具调用循环
  └── 探索 stream 和 interrupt（人在回路）

第 3 步：langchain（智能体）
  ├── 使用 create_agent() 快速构建智能体
  ├── 掌握 middleware 系统（如何添加自定义中间件）
  ├── 使用结构化输出（response_format）
  └── 理解 init_chat_model() 多提供商支持
```
